Introduction

This guide covers how to integrate a native app extension or WatchKit app built in Xcode with your Titanium application as well as information about sharing data between the extension and application. This guide does not cover how to develop an app extension or WatchKit app. Currently, the Titanium SDK supports iOS app extensions and WatchKit apps that are natively built in Xcode.

An iOS application extension allows your application to extend its functionality to other iOS applications. For example, when a document is displayed in the quick look previewer (Titanium.UI.iOS.DocumentViewer) and the user clicks the Share button, your Share or Action app extension appears in the list of options, which the user can select to send or share the data to your application.

A WatchKit app is a specialized app extension that runs on the Apple Watch. The WatchKit app allows the user to access data from your application on their Apple Watch.

Prerequisites

  • Titanium SDK 4.0 or greater

  • Xcode 6.2 or greater

  • Xcode Command-Line Tools

Create a Native App Extension

Use Xcode to create a native extension or Apple Watch app that can be integrated with your Titanium project. After the target is created, you will need to verify or change some settings, provision app group capabilities, then develop your application extension.

Create an Application Extension or WatchKit App Target

To create an application extension, you need to first create an Xcode project, then add your application extension targets.

  1. Open Xcode.

  2. Click Create a New Xcode project or from the menu bar, select File > New > Project.

  3. In the template dialog, select iOS > Application > Single View Application, then click Next.

  4. Change the Product Name and Organization Identifier, so the Bundle Identifier matches the Titanium App ID. For example, if you Titanium App ID is com.foo.test, the product name will be test and the organization identifier will be com.foo. This is not required but will make the subsequent steps easier.

  5. Click Next, then Create. The Xcode project will be created.

  6. From the menu bar, select File > New > Target.

  7. Select either iOS > Application Extension, then the extension you want to create; or iOS > Apple Watch > WatchKit App to create an Apple Watch app.

  8. Enter a product name for the extension. This name will be used to identify the extension to the Titanium application. For WatchKit, the name of the extension will be <PROJECT_NAME> WatchKit Extension.

  9. Click Next, then Finish.

  10. When prompted to create a scheme, click Activate.

A new target will be added to the Xcode project.

Verify Target Settings

In order to build the native app extension with a Titanium application, you need to verify or change certain target settings.

  1. Under General > Identity:

    1. The Bundle Identifier of the app extension must begin with the Titanium App ID it will be included in. For example, if the ID of the Titanium app is com.foo.test then the bundle ID of the extension may be com.foo.test.MyExt.

    2. Select your team from the Team drop-down. If the Fix Issue button is present, click it to fix any provisioning profile issues.

  2. Under Build Settings > Packaging, the Product Name of the app extension must be $(TARGET_NAME).

  3. Under Build Settings > Code Signing, the Code Signing Identity must be the same identity used to sign the Titanium application.

For WatchKit, you need to verify both the WatchKit extension and the WatchKit app. If you change the Bundle Identifier in the WatchKit app, you will also need to update:

  1. In the WatchKit App's Supporting Files/Info.plist file, change the WKCompanionAppBundleIdentifier to the Titanium App ID.

  2. In the WatchKit Extension's Supporting Files/Info.plist file, change the NSExtension > NSExtensionAttributes > WKAppBundleIdentifier to the WatchKit App's Bundle Identifier.

Provision an App Group

To share information or files between the application and extension, you need to provision an app group. An app group allows multiple applications to share the same file container and enables inter-process communication.

  1. In the target settings of the extension, select the Capabilities tab.

  2. Turn the App Groups switch to ON.

  3. When prompted, select your Development Team, then click Choose.

  4. If the Fix Issues button appears under the App Group section, click it.

  5. Click the Add button to create new app groups or enable existing app groups.

This procedure automatically creates the entitlements file and App ID for the extension. You will need to generate a provision profile from the App ID to run the extension on a device or to package it. You will also need to add the app group provisions to the Titanium application, which is described below.

Setup the Titanium Application

To use native app extensions with your Titanium application, you need to modify the tiapp.xml to include the extension when building them, provision the App ID to use the same app group(s) as the extension, then add the app group(s) to the Entitlements.plist file.

Add Extensions to the Project

Adding extensions is similar to adding modules using a text editor.

  1. Open the project's tiapp.xml file in an editor.

  2. Add the <extensions> tag as a child of the <ti:app> tag.

  3. For each app extension, add an <extension> tag with the projectPath attribute set to the absolute path to the folder containing the Xcode project and the target folders, and the node text of the tag contains the Target/Product Name of the extension. For WatchKit, the name of the extension will be <XCODE_PROJECT_NAME> WatchKit Extension.

For example, if your Xcode project is located at /Users/Shared/Xcode_Projects/MyExt/ and the name of the extension is ActionExt, add the following content to the tiapp.xml file:

<ti:app>
<extensions>
<extension projectPath="/Users/Shared/Xcode_Projects/MyExt/">ActionExt</extension>
</extensions>
</ti:app>

Add an App Group

To add an app group to the application, you must provision the app group capability in the App ID, then add the app group.

When creating the App ID, enable the App Group service.

Then, you add app groups to the App ID.

  1. Click the App ID you want to add app groups to.

  2. At the bottom of the App ID, click the Edit button.

  3. If the App Group service was not previously enabled, enable it.

  4. Click the Edit button next to the App Group service.

  5. Select the App Groups you want to add, then click Continue.

  6. Click Assign.

Generate or regenerate your provision profile(s).

Add an Entitlements.plist File

Create an Entitlements.plist file at the root of the project folder (same level as the Resources folder) or build the project once to generate a default one in the build/iphone folder, then copy it to the root level of the project. In the <dict> element, add the com.apple.security.application-groups key where the value is an array of string values for each app group name.

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
<key>com.apple.security.application-groups</key>
<array>
<string>group.foo.share</string>
</array>
</dict>
</plist>

Share Data Between the Application and Extension

To share data between the Titanium application and the app extension, you need to provision the App Group capability and add the same app group to each App ID and Entitlements.plist file (the Titanium application and app extension) as mentioned previously.

Titanium exposes two ways to share data:

  1. Read and write properties using the Titanium.App.iOS.UserDefaults API for the Titanium application and the NSUserDefaults API for the native app extension

  2. Read and write files to a common shared directory (called a "container" in iOS)

The community has developed modules to help exchange information between your extension and Titanium application. The Ti.Wormhole module, developed by Titanium Ben Bahrenburg, is an example of this that enables event support.

User Defaults

User defaults stores user preferences that are accessible by applications that use the same app group. User defaults works similarly to Ti.App properties except you need to create an instance using the Titanium.App.iOS.createUserDefaults() method and pass the method an object with only the suiteName property set to the app group you want to access. Once the instance is created, you can read and write properties to it.

var props = Ti.App.iOS.createUserDefaults({
suiteName: 'group.foo.game'
});
var score = props.getInt('score');
score += 50;
props.setInt('score', score);

In the extension, you can access the properties using the NSUserDefaults class.

NSUserDefaults *defaultsObject = [[NSUserDefaults alloc] initWithSuiteName:@"group.foo.game"];
NSNumber *score = [NSNumber numberWithDouble:[defaultsObject doubleForKey:@"score"]];
NSLog(@"%@", score);
[self.label setText:[score stringValue]];

Shared Directory

Use the Titanium.Filesystem.directoryForSuite() method to retrieve the shared location where the applications can access files. Pass the method the app group name.

// Get the shared container path
var sharedPath = Ti.Filesystem.directoryForSuite('group.foo.game');
 
// Copy a file from the application to the shared container
var appFile = Ti.Filesystem.getFile(Ti.Filesystem.applicationDataDirectory, 'reward.txt');
var sharedFile = Ti.Filesystem.getFile(sharedPath, 'reward.txt');
sharedFile.createFile();
sharedFile.write(appFile.read());
 
// Create a database accessible in the shared container
var db = Ti.Database.open(sharedPath + 'history');
db.close();

In the extension, to retrieve an NSURL of the share location, use the [NSFileManager containerURLForSecurityApplicationGroupIdentifier:] method.

Run the Project

To run the project, just build and launch the project like you would any other Titanium project. For Studio, use the run command from the launch options toolbar, and for the CLI, exectue appc run -p ios with the additional build options.

For each App ID that Xcode creates for the each extension, you need to generate a provision profile. Alternatively, you can use a wildcard App ID that enables the App Group capability. To generate provision profiles for development builds, see Deploying to iOS devices.

For the WatchKit app, the Appcelerator CLI and Studio currently do not launch the WatchKit app in the simulator. After the Titanium application launches, you need to manually run the following command from the terminal to launch the WatchKit app:

xcrun simctl launch booted <WATCHKIT_APP_BUNDLE_IDENTIFIER>
 
## Example
xcrun simctl launch booted com.foo.extension.watchkitapp

If you get a black screen after the loading screen in the Watch simulator but did not get an error when executing the command, try re-running the xcrun command. You may also get a black screen on the iOS simulator after the command executes. Go back to the Home screen and relaunch the application.

Package the Application

To package the application with the extension, just package the project like you would any other Titanium application. For Studio, use the package command from the launch options toolbar, and for the CLI, execute appc run command with the package options.

For each App ID that Xcode creates for each extension, you need to generate a provision profile. Alternatively, you can use a wildcard App ID that enables the App Group capability. To generate provision profiles for production builds, see Distributing iOS apps.

For WatchKit apps, you need to perform some additional steps:

  1. Build the Titanium app for the App Store.

  2. In the Xcode Organizer, right- or control-click on the Archive and click Show in Finder.

  3. In the Finder, right- or control-click on the .xcarchive file and click Show Package Contents.

  4. Download and unzip the WatchKitSupport.zip file and add it to your .xcarchive as seen here:

  5. Submit the archive to the App Store.

Troubleshooting

Project Issues

  • Build the targets in Xcode to make sure the extension or WatchKit app works correctly.

  • If you have multiple versions of Xcode, verify that xcode-select references an Xcode with version 6.2 or greater. Execute xcode-select -p to see which Xcode is being used, then launch it to verify the version.

  • If you encounter the SPErrorInValidBundleNoGizmoBinaryMessage error and changed the Bundle Identifier of the WatchKit app, you also need to update the Info.plist files in the WatchKit app and extension as instructed in Verify Target Settings.

  • If you made changes in the Xcode project and do not see them when running the Titanium project, try cleaning both the extension target (from the Xcode menu bar, select Product > Clean) and the Titanium application (from the Studio menu bar, select Project > Clean), then rebuild the project.

Simulator Issues

  • If you see a black screen in the Watch simulator, execute (or re-execute) the xcrun command as instructed in Run the Project. Also verify that your WatchKit app builds and runs correctly from Xcode.

  • If you see a black screen in the iOS simulator after executing the xcrun command, go back the to Home screen and relaunch the application.

  • Try resetting the iOS simulator. From the menu bar, select iOS Simulator > Reset Contents and Settings....

Package Issues

  • If you get validation errors regarding an "Invalid Binary" when submitting the package to the iTunes App Store, make sure you copy the unzipped WatchKitSupport folder to your project archive as instructed in Package the Application.
